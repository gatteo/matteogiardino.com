Eseguire modelli AI localmente significava privacy totale, ma il panorama sta cambiando. Le vulnerabilità non risolte di Ollama stanno trasformando i setup locali in potenziali vettori d'attacco. Ti spiego come funzionano il phishing e l'esfiltrazione dati tramite modelli malevoli e, soprattutto, come mettere in sicurezza il tuo sistema.

Quando scarichi un modello dalla repository di Ollama, non stai semplicemente ottenendo dei pesi matematici; stai eseguendo file di configurazione e potenziali logiche di strumenti esterni. Se ti fidi ciecamente dei modelli della community senza controlli sul traffico in uscita, il tuo ambiente di sviluppo è a rischio.

## Il falso senso di sicurezza locale

Vedo molti sviluppatori configurare un'istanza Ollama sul proprio Mac Mini, pensando che "locale" significhi automaticamente "sicuro". La realtà è che le vulnerabilità nel modo in cui Ollama gestisce le interazioni di sistema e i tool call hanno creato enormi falle nella sicurezza.

I modelli ospitati su registri aperti possono includere configurazioni `Modelfile` malevole. Quando il tuo agente locale interagisce con questi modelli, potrebbe inavvertitamente eseguire comandi che inviano le tue variabili d'ambiente o file locali a server esterni.

## Come funziona il Phishing in Ollama

Il phishing nel contesto dell'AI locale non si presenta come un'email che ti chiede la password. Invece, un modello malevolo potrebbe richiedere le chiavi API all'utente simulando un messaggio di "autenticazione richiesta" direttamente nel prompt di sistema.

Poiché gli utenti si fidano del proprio terminale o client AI locale, sono molto più propensi a fornire le credenziali. Il modello utilizza poi un tool o una richiesta di rete (se il traffico in uscita non è bloccato) per esfiltrare quella chiave.

## Rischi di esfiltrazione dati tramite Tool Call

Con la recente introduzione dell'uso degli strumenti (function calling) nei modelli locali, la superficie d'attacco si è allargata. Un utente malintenzionato può creare un modello che altera sottilmente le chiamate alle funzioni previste. Se il tuo framework esegue automaticamente queste chiamate senza una rigorosa validazione, il modello può leggere file locali sensibili e inviarli all'esterno.

Se non limiti l'accesso di rete del tuo container Ollama, qualsiasi processo in esecuzione al suo interno può comunicare con l'esterno.

## Come proteggere il tuo setup AI locale

La sicurezza nell'AI locale richiede un approccio su più livelli. Non puoi fare affidamento solo sul framework.

1. **Filtra il traffico in uscita:** Non permettere mai ai tuoi agenti o processi Ollama di accedere liberamente a internet. Usa un proxy per limitare il traffico in uscita solo verso endpoint conosciuti.
2. **Esamina i Modelfile:** Prima di eseguire un modello della community, controlla il suo `Modelfile`. Cerca prompt di sistema anomali o modelli di base sconosciuti.
3. **Isola il tuo ambiente:** Fai girare i tuoi carichi di lavoro AI in container o macchine virtuali dedicate, senza accesso al tuo filesystem principale.
4. **Usa solo strumenti approvati:** Inserisci in una whitelist esplicita gli strumenti che i tuoi agenti possono utilizzare.

Trattando gli LLM locali con la stessa cautela che riservi alle API esterne, puoi mitigare efficacemente queste vulnerabilità non risolte.

## Domande frequenti

**I modelli ufficiali di Ollama sono sicuri?**
Sì, i modelli pubblicati sotto i namespace ufficiali sono generalmente sicuri. Devi fare attenzione ai modelli caricati dagli utenti della community.

**Usare Ollama in Docker risolve il problema?**
No. Docker fornisce isolamento, ma se il container ha accesso illimitato a internet in uscita, l'esfiltrazione dei dati è ancora possibile.

**Come posso bloccare le richieste in uscita non autorizzate?**
Usa uno strumento come Envoy o un firewall di sistema per implementare una whitelist, bloccando tutto il traffico proveniente dal container Ollama tranne per i domini esplicitamente approvati.
