## Comprendere i limiti di Gemini 3.1 Pro

Antigravity impone rigidi limiti di token al minuto (TPM) e richieste al minuto (RPM) quando si usa Gemini 3.1 Pro. Sebbene Gemini offra una finestra di contesto enorme, raggiungere il limite RPM è sorprendentemente facile se i tuoi agenti sono molto attivi.

## Soluzione 1: Implementare strategie di Backoff

La prima linea di difesa è implementare una strategia di backoff esponenziale nel tuo codice. Quando Antigravity restituisce un 429, il tuo script dovrebbe attendere qualche secondo prima di riprovare, raddoppiando il tempo di attesa ai tentativi successivi.

```python
import time
import requests

def call_gemini_with_backoff(prompt, max_retries=5):
    for attempt in range(max_retries):
        response = requests.post("https://api.antigravity.example/v1/chat/completions", json={"model": "gemini-3.1-pro", "messages": [{"role": "user", "content": prompt}]})
        if response.status_code == 200:
            return response.json()
        elif response.status_code == 429:
            time.sleep(2 ** attempt)
        else:
            raise Exception(f"Errore API: {response.status_code}")
    raise Exception("Tentativi massimi superati")
```

## Soluzione 2: Batching delle richieste

Invece di inviare molte piccole richieste, raggruppa i tuoi prompt. Gemini 3.1 Pro eccelle nel seguire istruzioni complesse e multi-parte. Combina i compiti paralleli del tuo agente in un singolo blocco e chiedi al modello di restituire un JSON strutturato.

## Soluzione 3: Fallback Routing

## FAQ: Quote Antigravity per Gemini 3.1 Pro

**Quali sono le quote antigravity per gemini 3.1 pro?**
Le quote antigravity per gemini 3.1 pro si riferiscono ai rigorosi limiti di velocità imposti quando si fa il proxy delle richieste Gemini, tipicamente limitati a specifici livelli RPM e TPM.

**Come posso evitare l'errore 429?**
Implementa il backoff esponenziale, raggruppa le attività dell'agente in un singolo prompt e memorizza nella cache le richieste ricorrenti.
