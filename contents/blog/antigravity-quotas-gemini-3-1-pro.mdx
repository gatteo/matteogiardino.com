## Understanding the Gemini 3.1 Pro Limits

Antigravity imposes strict tokens-per-minute (TPM) and requests-per-minute (RPM) limits when proxying Gemini 3.1 Pro. While Gemini itself offers a massive context window, hitting the RPM limit is surprisingly easy if your agents are highly active.

## Workaround 1: Implementing Backoff Strategies

The first line of defense is implementing an exponential backoff strategy in your client code. When Antigravity returns a 429, your script should wait a few seconds before retrying, doubling the wait time on subsequent failures.

```python
import time
import requests

def call_gemini_with_backoff(prompt, max_retries=5):
    for attempt in range(max_retries):
        response = requests.post("https://api.antigravity.example/v1/chat/completions", json={"model": "gemini-3.1-pro", "messages": [{"role": "user", "content": prompt}]})
        if response.status_code == 200:
            return response.json()
        elif response.status_code == 429:
            time.sleep(2 ** attempt)
        else:
            raise Exception(f"API Error: {response.status_code}")
    raise Exception("Max retries exceeded")
```

## Workaround 2: Request Batching and Token Optimization

Instead of sending many small requests, batch your prompts. Gemini 3.1 Pro excels at following complex, multi-part instructions. Combine your agent's parallel tasks into a single prompt block and ask the model to return a structured JSON response covering all tasks at once.

## Workaround 3: Fallback Routing

When your Antigravity Gemini 3.1 Pro quota is completely exhausted, route non-critical tasks to a cheaper, faster model like Gemini 3.1 Flash or Claude 3.5 Haiku. Only reserve the Pro model for heavy reasoning tasks.
