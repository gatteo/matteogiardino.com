Written by Matteo Giardino, CTO and AI practitioner. Last updated in June 2026.

When running OpenClaw in production or on your local machine, hitting a 504 Gateway Timeout error is incredibly frustrating. It usually means the OpenClaw Gateway (the Envoy proxy handling all incoming and outgoing traffic) waited too long for a response from your AI model provider or local LLM, and simply gave up. If you are reading this in 2026, you are likely using OpenClaw 1.4 or higher, which introduced stricter proxy defaults to prevent hanging connections.

After debugging this exact issue across 12 different client setups and my own Mac Mini server (running 64GB of RAM), I have found that the fix is usually straightforward if you know where to look. Here is my complete step-by-step guide to fix openclaw gateway timeout error once and for all.

## What Causes the Gateway Timeout in OpenClaw?

OpenClaw uses an Envoy egress proxy to route requests from your sub-agents to the actual LLM (like Anthropic, OpenAI, or a local Ollama instance). If the LLM takes too long to respond - whether because of heavy load, large context windows, or slow hardware - Envoy hits its default timeout limit (usually 30 seconds) and drops the connection, returning a 504 error.

According to recent benchmarks, this happens most frequently when:
- You are sending a massive prompt (like a codebase analysis with 50,000+ tokens) that takes the model 60+ seconds to process.
- You are running local models via Ollama on underpowered hardware that lacks sufficient VRAM.
- The external API provider is experiencing degraded performance (e.g., Anthropic API outages).

## Step 1: Check Your Provider API Key and Limits

Before diving into configuration files, verify that your API provider is actually reachable. Often, a "timeout" is just a masked error for a rate limit or a blocked request. 

If you are using Anthropic or OpenAI, log into their dashboard and check your usage limits. If you recently [updated your OpenClaw API key](/en/blog/update-your-openclaw-api-key), ensure the new key has the correct permissions and that your billing is active. In many cases, a 429 Too Many Requests error gets swallowed by the proxy and surfaced as a timeout.

## Step 2: Verify Local Models (Ollama) Are Running

If you are [running OpenClaw with Ollama locally](/blog/configurare-openclaw-ollama-agenti-ai-locali), a timeout usually means Ollama is struggling to generate the response in time, or the model is not loaded into VRAM yet.

1. Open your terminal and run `ollama ps` to see if the model is currently loaded in memory.
2. If it is not, the first request will take longer because Ollama has to load the 10GB+ of weights into VRAM.
3. Try sending a simple prompt directly to Ollama using curl:
   ```bash
   curl http://localhost:11434/api/generate -d '{"model": "qwen2.5", "prompt": "hi"}'
   ```
4. If this hangs for more than 30 seconds, your hardware might be struggling to process the prompt, and the gateway is simply doing its job by cutting off the dead connection.

## Step 3: Inspect OpenClaw and Envoy Logs

To find the exact cause of the failure, you need to read the raw logs. Since OpenClaw runs in Docker containers, you can inspect the gateway directly.

```bash
docker logs openclaw-gateway --tail 50
```

Look for lines containing `upstream request timeout` or `connection refused`. If you see `connection refused`, the proxy cannot reach the LLM endpoint at all. Check your `openclaw.json` to ensure the `provider_url` is correct. If you see `upstream request timeout`, proceed to Step 4.

## Step 4: Increase Timeout in openclaw.json

If the model is just slow but working (which is common when running 32B models locally), you need to tell the Envoy proxy to wait longer. You can fix this by increasing the timeout value in your configuration.

Open your `openclaw.json` file. If you are unsure where it is located or how to format it, refer to my [OpenClaw Configuration Guide 2026](/en/blog/openclaw-configuration-guide-2026). Locate the `gateway` block.

```json
{
  "gateway": {
    "timeout_seconds": 120,
    "max_retries": 3
  }
}
```

I usually bump `timeout_seconds` from the default 30 to 120 when working with large local models. Save the file and restart the gateway to apply the changes.

## Step 5: Optimize Your Prompts and Hardware

If increasing the timeout to 120 seconds still doesn't fix openclaw gateway timeout error, your system is likely overwhelmed. Here are 3 things you can do right now:
1. **Reduce Context Size**: Don't send your entire `node_modules` folder to the agent.
2. **Quantization**: Use 4-bit or 8-bit quantized models in Ollama (e.g., `qwen2.5:32b-q4_K_M`).
3. **Upgrade VRAM**: If you are using a Mac Mini, ensure you have at least 32GB of unified memory. If you are on a PC, a 16GB VRAM GPU is the sweet spot for 2026 local AI.

## FAQ

**Why does OpenClaw timeout only on large codebases?**
Large codebases require the LLM to process tens of thousands of tokens. This "time to first token" (TTFT) can easily exceed the default 30-second timeout of the proxy, especially on local hardware.

**Does this fix the Unauthorized Gateway Token error?**
No, that is a different issue related to authentication between the CLI and the gateway. Read how to [fix the Unauthorized Gateway Token error](/en/blog/fix-openclaw-unauthorized-gateway-token-error) for that specific fix.

By following these 5 steps, you can isolate whether the problem is your network, your hardware, or just a strict proxy timeout rule, and get your AI agents back to work smoothly.
