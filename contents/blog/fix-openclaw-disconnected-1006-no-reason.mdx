If you're seeing the `Error: gateway closed (1006 abnormal closure (no close frame)): no close reason` or `disconnected (1006): no reason` in OpenClaw, you're not alone. I've encountered this issue multiple times on my own servers, and the fix is usually straightforward depending on your setup. In this guide, I'll walk you through the actual solutions that work.

## What OpenClaw Error 1006 Means

WebSocket error 1006 simply means the connection was closed abnormally. The server didn't provide a proper close frame or a clear reason for the disconnect. 

In the context of OpenClaw or Clawdbot, this usually indicates one of a few common problems: the gateway on `127.0.0.1:18789` isn't reachable, WebSocket upgrades are failing at your proxy level, a firewall is blocking traffic, or a Docker container has crashed. Let's fix it.

## Fix 1: Nginx WebSocket Headers (Most Common)

If you are running OpenClaw behind an Nginx reverse proxy on a VPS, this is almost certainly the cause. WebSockets require special headers to upgrade the connection. Without them, the gateway disconnects with a 1006 error immediately.

Make sure your Nginx configuration block includes the correct `Upgrade` headers:

```nginx
location / {
    proxy_pass http://127.0.0.1:18789;

    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";

    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;

    proxy_read_timeout 3600;
    proxy_send_timeout 3600;
}
```

After updating the config, reload Nginx:

```bash
sudo systemctl restart nginx
```

Reload OpenClaw, and the connection should stay open.

## Fix 2: Ubuntu/Debian Firewall Rules

Sometimes the gateway is actually running perfectly fine, but firewall rules are blocking the traffic on the default port. 

You need to allow both TCP and UDP traffic on the Web UI port:

```bash
sudo iptables -A INPUT -p tcp --dport 18789 -j ACCEPT
sudo iptables -A INPUT -p udp --dport 18789 -j ACCEPT
```

Then, force restart the gateway using the CLI:

```bash
clawdbot gateway --force
```

Keep in mind that iptables rules aren't persistent after a reboot unless you explicitly save them using `netfilter-persistent` or `iptables-save`.

## Fix 3: Docker Container State and Restarts

If you are deploying OpenClaw via Docker, check the container status. A stale or crashed container will cause repeated 1006 closures.

First, list your containers:

```bash
sudo docker ps
```

If you see your OpenClaw container in the list (even if it appears running but is stuck), restart it:

```bash
sudo docker restart <container_name>
```

Refresh the Web UI and wait a few seconds after login. This simple restart clears out stale WebSocket states.

## Fix 4: Disabling Telegram Channels

Multiple users have reported that the 1006 issue began immediately after configuring the Telegram channel integration. If you suspect this, you can disable it to see if the gateway stabilizes.

Run the configuration command:

```bash
clawdbot configure
```

Navigate to **Channels** and temporarily disable Telegram (or all channels). Once disabled, the gateway often remains stable. Unstable network routing to Telegram APIs can sometimes tear down the entire gateway connection. If you need Telegram, ensure your server can reach `api.telegram.org` reliably. 

## Conclusion & Next Steps

Error 1006 in OpenClaw looks intimidating but is almost always a networking layer issue. By fixing your Nginx upgrade headers, opening the correct firewall ports, or resolving Docker state issues, you'll be back up and running.
